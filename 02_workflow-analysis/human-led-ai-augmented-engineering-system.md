# 人主导的 AI 增强工程系统（Human-Led, AI-Augmented Engineering System）

## 1. 文档定位

本文是对 `knowledge-and-execution-dual-loop.md` 讨论的进一步综合结论，整合两轮 AI 分析：

1. 第一轮：提出“Web AI + Local Asset”的分层方向
2. 第二轮：在质疑基础上修正为“弱结构 + 强人控”的落地原则

目标是形成可执行、不过度设计、可长期演进的工程方法。

## 2. 综合结论

核心判断如下：

- AI 应作为信息转换增强器，而非流程主导者
- 资产应沉淀在本地工程环境，作为长期记忆与复用基础
- 显式结构应按需触发，而非默认强制
- 人必须保留关键认知节点的决策权

一句话表达：

`AI 是加速器，不是驾驶员；结构是工具，不是约束；人是系统调度核心。`

## 3. 基础架构分层

### 3.1 两层基础架构

```text
Web ChatGPT        = Knowledge + Bridge 层（信息转换）
Local IDE + Repo   = Execution + Asset 层（执行与沉淀）
```

这对应双循环模型中的两类核心机制：

1. AI：信息转换机制
2. Asset：信息沉淀机制

### 3.2 三层完整架构（推荐）

在两层基础上补足“桥接层”：

```text
1) AI 转换层（Web）
2) 桥接层（Context / History / Task）
3) 执行与资产层（Local IDE / Repo）
```

桥接层的作用是降低人工复制粘贴导致的上下文丢失与吞吐瓶颈。

## 4. 关键矛盾与设计立场

实践中存在一组核心张力：

- 结构化（显式） vs 流畅性（低负担）
- AI 自动化 vs 人的理解与控制

最终立场：

`优先保证“人对系统的理解权与决策权”，再追求自动化。`

这不是保守，而是长期可持续的工程设计。

## 5. 角色模型与分工

```text
Human = 决策者（Decision Maker）
AI    = 转换器（Transformer）
System= 执行器（Executor）
```

职责边界：

### 5.1 Human 必须主导

- 问题是否成立（Problem Framing）
- 方案是否可信（Decision）
- 是否沉淀为资产（Abstraction）

### 5.2 AI 适合增强

- 总结、改写、对比
- 方案候选枚举
- 日志归因与排查路径
- 代码生成与修改建议

### 5.3 System 负责执行

- IDE/Repo 操作承载
- 构建、测试、发布等自动化执行
- 资产版本化与追踪

## 6. 从“隐式桥接”到“显式桥接”

### 6.1 当前常见形态（隐式）

```text
IDE 报错 -> 复制到 Web -> AI 分析 -> 回 IDE 执行
```

问题：

- 切换成本高
- 信息丢失
- 重复描述
- 吞吐受限

### 6.2 目标形态（显式）

```text
IDE -> 记录问题上下文 -> AI 分析 -> 生成任务 -> IDE 执行 -> 资产沉淀
```

注意：显式化不是强制流程，而是按需策略。

## 7. 落地原则：弱结构 + 强人控

### 7.1 反对的形态

- 强制每次写 `problem.md / task.md / asset.md`
- 流程先于思考
- 人变成流程执行器

### 7.2 推荐的形态

- 结构是工具，可启可停
- AI 可用可不用
- 关键节点必须人判断

即：

`需要时显式化，不需要时保持流畅。`

## 8. 触发机制：按认知压力显式化

### 8.1 低复杂度（绿色）

- 直接问 AI
- 解决后结束

### 8.2 中复杂度（黄色）

- 先写几句上下文
- 再让 AI 分析

### 8.3 高复杂度（红色）

- 多轮失败或反复出现
- 引入最小结构化记录：
  - `problem.md`
  - `task.md`
  - `asset.md`

## 9. 最小可执行实现（MVP）

建议在仓库内建立轻量桥接目录：

```text
/engineering-os/
  /notes/
```

初始只维护三类记录：

1. Problem：现象、上下文、本质猜测
2. Task：执行步骤与验证方式
3. Asset：结论、可复用点、适用边界

实践要求：

`先记录再标准化，先复用再体系化。`

## 10. 对 engineering-os 的意义

这一路径避免两个极端：

1. AI 至上：自动化过度导致能力退化
2. 工程洁癖：过度流程导致效率下降

目标形态是：

`人 + AI 协同进化系统`

即在效率提升与认知成长之间建立长期平衡。

## 11. 结论

基于双循环讨论的最终表述是：

- 软件开发是知识工作与工程工作的耦合系统
- AI 负责跨循环信息转换，Asset 负责长期沉淀
- 人必须主导关键认知节点
- 结构应按认知压力触发，而非流程化强制

最终方法：

`拆分小循环，用 AI 做转换，用资产做沉淀，以人作为闭环调度者。`
