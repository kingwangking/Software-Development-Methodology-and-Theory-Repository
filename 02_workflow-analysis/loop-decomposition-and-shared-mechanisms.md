# 多循环拆分与共享机制（Loop Decomposition and Shared Mechanisms）

## 1. 文档目的

本文基于“知识-执行双循环模型”进一步提出一种更可落地的工程方法：

- 不先构建完整流程系统
- 先拆分真实工作中的小循环单元
- 再用共享机制连接循环

目标是降低流程设计复杂度，避免过度设计，并提升 AI 协同开发的可执行性。

## 2. 问题定义

在实践中，开发者看到的流程常常是线性的：

`Problem -> Search -> Learn -> Edit -> Build -> Run -> Test -> Debug -> Document`

但实际体验通常是循环嵌套：

`Edit -> Build -> Error -> Search -> Fix -> Build -> Test -> Debug -> Search -> ...`

这说明真实开发并不是“一条大链路”，而是“多个小循环交织运行”。

## 3. 方法原则

本文采用两步法：

1. 循环单元拆分（Loop Decomposition）
2. 共享机制连接（Shared Mechanisms Integration）

这是一种从“流程建模”转向“行为原子 + 共享机制建模”的方式。

## 4. 四类核心循环

### 4.1 问题理解循环（Knowledge Loop）

`Problem -> Search -> Read -> Understand -> Search`

特征：

- 主要发生在知识环境
- 目标是形成认知与判断
- 结果是“可执行理解”，而非代码产出

### 4.2 实现验证循环（Execution Loop）

`Edit -> Build -> Run -> Test -> Debug -> Edit`

特征：

- 主要发生在执行环境
- 目标是让系统可运行、可验证
- 结果是代码与行为正确性

### 4.3 问题驱动桥接循环（Bridge Loop）

`Error/Uncertainty -> Search -> Apply -> Execute Again`

特征：

- 横跨知识环境与执行环境
- 触发频率高
- 是开发过程中的关键吞吐瓶颈

### 4.4 经验沉淀循环（Asset Loop）

`Solve -> Summarize -> Record -> Reuse`

特征：

- 常被忽略但价值长期累积
- 决定团队是否“重复踩坑”
- 产出可复用工程资产

## 5. 共享机制：跨循环连接点

跨循环真正的重合点不是“流程步骤相同”，而是“机制可共享”。

### 5.1 机制一：信息转换（AI Mechanism）

AI 的核心作用是信息形态转换：

- 问题 -> 可搜索表达
- 搜索结果 -> 可理解总结
- 理解 -> 可执行步骤
- 错误日志 -> 可能原因与排查路径
- 经验结论 -> 可复用模板

因此 AI 更适合被定义为：

`Cross-loop Information Transformer`

### 5.2 机制二：信息沉淀（Asset Mechanism）

资产管理的本质是：

`将瞬时认知转化为长期可复用结构`

沉淀对象包括：

- 问题定义
- 关键判断
- 解决方案
- 可复用约定/模板

## 6. 重组后的流程观

建议采用以下上层模型：

```text
多个小循环（Knowledge / Execution / Bridge / Asset）

         ↓ 通过共享机制连接

1) AI：信息转换
2) Asset：信息沉淀
```

该模型下，流程不再是主体，机制才是跨场景复用主体。

## 7. 最小可执行落地方案

### 7.1 AI 使用模式优化（而非工具堆叠）

在不同循环节点使用不同提示策略：

- Knowledge Loop：问题澄清、方案对比、概念解释
- Bridge Loop：错误归因、排查路径、验证顺序
- Execution Loop：代码审查、测试建议、重构建议

原则：

`AI = 分析器 / 转换器，不是答案机`

### 7.2 极简资产记录（而非先建复杂系统）

建议从 `/notes/` 开始，只记录三类资产：

1. 问题（现象、上下文、本质）
2. 关键结论（决策与约束）
3. 可复用方案（模板、脚手架、清单）

原则：

- 先记录再标准化
- 先复用再体系化
- 让模板在实践中自然生长

## 8. 阶段判断

当前阶段不应优先“工具建设”，而应优先“工作方式迁移”。

即：

- 从线性流程思维迁移到多循环思维
- 从人工串行执行迁移到 AI 转换增强 + 资产沉淀闭环

## 9. 对 engineering-os 的启示

engineering-os 的最小闭环应围绕两类跨循环能力建设：

1. AI 信息转换能力（问题到执行）
2. 资产沉淀能力（执行到知识）

推荐建设顺序：

1. 明确循环边界与输入输出
2. 在桥接循环优先接入 AI
3. 建立轻量知识资产目录
4. 再逐步抽象为标准流程与系统能力

## 10. 结论

不要先构建完整流程系统，而应：

`拆分多个小循环，用 AI 做转换，用资产做沉淀。`

这一路径更符合真实开发行为，也更容易形成可持续的工程能力。
