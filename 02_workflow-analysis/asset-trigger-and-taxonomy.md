# 资产沉淀判定与分类（Asset Trigger and Taxonomy）

## 1. 文档目的

本文用于判断“当前问题或结论是否值得沉淀为工程资产”，并给出资产分类方式，帮助团队在效率与长期复用之间取得平衡。

核心目标：

- 避免“为记录而记录”
- 避免“有价值经验未沉淀”
- 让资产从偶然产物变成可管理产物

## 2. 适用范围

适用于 `engineering-os` 语境下的资产沉淀决策，尤其适用于以下循环的输出：

- Knowledge Loop
- Bridge Loop
- Asset Loop

可用于 `problem/task/asset` 相关文档及其后续结构化归档。

## 3. 核心原则

1. 只在认知收敛时沉淀
2. 只沉淀可复用内容
3. 不为记录而记录

补充原则：

- 人主导判定，AI 仅辅助归纳
- 先轻量记录，再逐步标准化
- 资产必须可追溯到问题背景

## 4. 触发信号（建议沉淀）

当出现以下任一情况时，建议沉淀为资产：

1. 多轮讨论后形成稳定结论
2. 同类问题重复出现（第二次及以上）
3. 形成了明确判断原则或设计约束
4. 解决方案具备明显复用价值
5. 可抽象为模板、清单、模式或规则
6. 该结论会影响后续决策一致性

## 5. 非触发场景（不建议沉淀）

以下情况通常不建议沉淀：

1. 一次性问题且无复用价值
2. 纯工具操作步骤且无抽象结论
3. 尚未验证或分歧较大的结论
4. 仅信息收集，未形成判断
5. 沉淀成本明显高于潜在复用收益

## 6. 快速判定清单

在沉淀前，优先回答三个问题：

1. 未来是否大概率再次遇到此类问题？
2. 当前是否已经形成“判断”，而不仅是“信息”？
3. 下次遇到时，是否希望直接复用这份内容？

若至少 2 项回答为“是”，建议沉淀。

## 7. 一句话触发规则

当一个问题从“这次怎么解决”升级为“以后这类问题应如何处理”，就应沉淀为资产。

## 8. 最小判定流程

```text
Solve
  -> Check Convergence
  -> Check Reusability
  -> Decide: Record or Skip
```

推荐执行逻辑：

1. 先完成问题解决
2. 再判断是否认知收敛
3. 再判断复用价值
4. 最后决定沉淀级别（不沉淀 / 轻量沉淀 / 标准沉淀）

## 9. 沉淀级别（建议）

### 9.1 Level 0：不沉淀

适用：一次性问题、无复用价值。

### 9.2 Level 1：轻量沉淀

适用：中等复杂度、可能复用。

形式：

- 短记录（几行结论 + 适用条件）

### 9.3 Level 2：标准沉淀

适用：高复用价值或影响决策一致性。

形式：

- 问题背景
- 判定依据
- 方案与边界
- 可复用模板/清单

## 10. 资产分类规则（Asset Taxonomy）

建议分类目录：

```text
/assets/
  /principles/   # 原则与判定标准
  /patterns/     # 可复用模式
  /solutions/    # 典型问题解决方案
  /templates/    # 直接可套用模板
```

分类建议：

1. `principles`：高稳定、低频变化、指导决策
2. `patterns`：跨场景可复用的结构化做法
3. `solutions`：与特定问题类型绑定的实践结果
4. `templates`：可直接执行或填写的操作框架

## 11. 治理与演进建议

1. 不追求一次性完美分类
2. 每月回看高频资产，做合并与去重
3. 资产更新需保留版本与变更说明
4. 对失效资产标注“过期/替代路径”
5. AI 可辅助整理，但最终发布由人审核

## 12. 与人机协同模型的关系

本规则与 `human-led-ai-augmented-engineering-system.md` 一致：

- Human：决定是否沉淀与如何抽象
- AI：辅助总结、改写、结构化
- System：承载版本化、检索与复用

因此，资产沉淀不是附属动作，而是人主导闭环中的核心治理能力。

## 13. 结论

资产沉淀的关键不在“写多少文档”，而在“是否在正确时机沉淀正确内容”。

方法上，应优先建立可执行判定规则，再逐步完善资产分类体系，使沉淀成为可控、可复用、可演进的系统能力。
