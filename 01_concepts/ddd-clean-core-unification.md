# DDD 与 Clean 架构的一体化理解：核心定义与核心保护

## 1. 文档目的

本文用于统一解释 DDD（领域驱动设计）与 Clean 架构之间的关系，回答两个问题：

1. 系统核心应由什么定义？
2. 结构应如何保证核心不被外层污染？

## 2. 核心结论

DDD 与 Clean 不是两套割裂的方法，而是一体两面：

- DDD 回答：`核心应该是什么`（问题导向的领域核心）
- Clean 回答：`有了核心以后如何组织结构`（依赖向核心收敛）

一句话：

`DDD 定义核心，Clean 保护核心。`

## 3. 为什么这两者能够统一

系统设计如果只停留在“分层”或“技术模块划分”，很容易退化为：

- Controller 中心
- Database 中心
- Framework 中心

而不是“问题中心”。

DDD 先把系统锚定到业务问题本身；
Clean 再通过依赖方向约束，把解释权固定在核心层。

## 4. 三个关键认识

### 4.1 系统必须先有核心

在讨论架构之前，先回答：

`这个系统到底围绕什么组织？`

若没有核心定义，后续分层与接口设计都会失去判断基准。

### 4.2 依赖倒置的本质是核心保护

依赖倒置不仅是“易测试、低耦合、可替换实现”的技巧，
其更本质的作用是：

`防止外部变化反向定义系统核心。`

即：

- 数据库不能定义领域模型
- Web 框架不能定义用例语义
- 输入输出形式不能定义业务对象
- 基础设施不能反向塑造核心规则

### 4.3 三层架构的问题是默认不保证核心

三层架构并非不能用，但其常见出发点是职责分层，而非核心建模。

典型风险：

- 业务层边界模糊
- 领域对象退化为数据对象
- 数据库结构反向决定系统结构
- 应用编排与领域规则混杂

更准确的说法：

- 三层架构可以解决工程分层
- 但天然不保证“问题核心”被建立与保护

## 5. 统一建模表达

可将二者统一为以下模型：

```text
Problem Domain
  -> Domain Core (DDD)
  -> Dependency Convergence (Clean)
  -> Stable System Behavior
```

其中：

- Domain Core：定义系统“是什么”
- Dependency Convergence：保证系统“不会被外层改写”

## 6. 实践判定清单

在架构评审时，可用以下问题快速检查是否真正“核心驱动”：

1. 领域概念是否独立于数据库表结构？
2. 用例语义是否独立于 Web/API 协议？
3. 领域规则是否可在不依赖框架的前提下表达？
4. 外层替换（DB/消息/框架）是否不改变核心语义？
5. 依赖方向是否持续朝向核心层收敛？

若多数为“否”，说明系统仍是外层驱动而非核心驱动。

## 7. 对架构设计的直接启示

1. 先做领域核心建模，再做技术分层。
2. 先定义领域语言与业务不变量，再定义接口与存储模型。
3. 把依赖倒置视为治理规则，而非编码风格。
4. 优先保护核心解释权，再优化外层工程效率。

## 8. 结论

DDD 与 Clean 的关系可以概括为：

- DDD 负责“系统应围绕什么建立”
- Clean 负责“系统如何持续围绕核心运行”

最终目标不是“分层正确”，而是：

`系统核心长期稳定、外层可替换、业务语义不失真。`
