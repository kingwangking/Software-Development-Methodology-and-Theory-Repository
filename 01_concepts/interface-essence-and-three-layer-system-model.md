# 接口的本质与系统三层模型

## 1. 文档目的

本文聚焦三个核心问题：

1. 接口（Interface）的本质是什么
2. 系统如何用“需求 -> 抽象 -> 实现”三层组织
3. 如何用这套认知重塑架构、分层、依赖与语言特性的理解

目标不是讲语法，而是建立稳定的设计判断框架。

## 2. 核心结论

接口不是为了“语法技巧”而存在，而是为了表达：

`系统需要什么能力边界与协作契约。`

因此：

- 需求层定义“要具备什么能力”
- 抽象层定义“能力如何被约定”
- 实现层负责“能力如何被兑现”

一句话：

`先定义能力，再选择实现。`

## 3. 接口的本质

常见说法是：

- 为了多态
- 为了解耦
- 为了依赖注入
- 为了测试替换

这些都成立，但它们更像“接口带来的效果”。

更本质的解释是：

`接口是问题空间与实现空间之间的契约表达。`

也就是说，接口首先服务于“能力定义”，其次才服务于“实现替换”。

## 4. 系统三层模型

```text
需求层（Problem / Requirement）
  -> 抽象层（Contract / Capability）
  -> 实现层（Technology / Execution）
```

### 4.1 需求层

关注：系统应该提供什么能力、遵守什么规则。

### 4.2 抽象层

关注：把能力定义成可协作的边界与契约。

### 4.3 实现层

关注：用数据库、框架、协议、设备等技术兑现抽象。

关键顺序：

`逻辑上抽象先于实现。`

（实际开发可先写实现再回抽象，但认知上应保持此顺序。）

## 5. 多方面例子

### 5.1 领域与仓储（DDD/Clean）

需求：系统需要“保存订单”。

抽象：`IOrderRepository` 定义“订单存取能力”。

实现：EF Core / Dapper / PostgreSQL 等具体实现。

流程使用者：Application Use Case 调用仓储能力完成用例。

### 5.2 支付能力

需求：系统需要“完成支付”。

抽象：`IPaymentGateway`（授权、扣款、回调验证）。

实现：Stripe / PayPal / 自研通道。

收益：业务流程不被具体支付厂商绑定。

### 5.3 时间能力

需求：系统需要“获取当前时间”。

抽象：`IClock`。

实现：系统时间、测试时钟、业务时区时钟。

收益：规则可测试，时间逻辑可控。

### 5.4 日志能力

需求：系统需要“记录运行行为”。

抽象：日志能力接口或应用层日志端口。

实现：Console / File / ELK / OpenTelemetry。

收益：监控能力可替换，业务逻辑不依赖具体日志基础设施。

### 5.5 高级语言特性

高级语言本身也是抽象层：

- `class`：对象职责抽象
- `interface`：协作契约抽象
- `async/await`：异步流程抽象
- LINQ：数据操作意图抽象

本质：把反复出现的问题结构提升到语言层表达。

## 6. 自然推导出的架构现象

当系统按“需求 -> 抽象 -> 实现”组织时，会自然出现：

- 依赖反转（内层依赖抽象而非外层实现）
- 多态替换（多个实现兑现同一契约）
- 依赖注入（装配手段，而非设计起点）
- 可测试性增强（契约可替换、行为可控制）

因此：

`依赖反转不是接口的目的，而是抽象先行后的结构结果。`

## 7. 用该认知重塑关键问题

### 7.1 重塑“接口要不要建”

判断问题应从“是否需要稳定能力边界”出发，而不是“是否可能替换实现”。

可用三问：

1. 这里是否存在稳定协作边界？
2. 这里是否在表达问题域中的角色能力？
3. 这里是否需要明确契约来约束流程协作？

若多数为“是”，接口有意义。

### 7.2 重塑“分层怎么划”

分层不是目录分组，而是解释权分配：

- Domain：定义“是什么”
- Application：定义“何时发生”
- Infrastructure：定义“如何实现”

### 7.3 重塑“技术选型优先级”

先问能力契约，再选技术实现；
避免“先选框架，再倒推业务结构”。

### 7.4 重塑“架构评审标准”

评审重点应从“用了多少模式”转向：

- 关键能力是否有清晰契约
- 契约解释权是否在内层
- 外层变化是否不会反向改写业务语义

## 8. 常见误区

1. 把接口当作必须普遍化的语法模板（到处 `IXXX`）。
2. 把接口仅理解成“方便未来替换实现”。
3. 以依赖注入框架为起点反向设计抽象。
4. 把局部实现细节误抽象为系统级契约。

## 9. 结论

你真正需要把握的不是“接口写法”，而是系统设计顺序：

`问题定义能力，抽象组织协作，实现兑现能力。`

接口位于问题与实现之间，是契约层的显式形态；
实现是接口的兑现者，而不是接口存在的理由。

最终原则：

`系统不是由实现直接拼起来的，而是由面向问题的抽象组织起来，再由实现填充。`
