# 问题导向与需求导向：软件系统建模深度的判断原则

## 1. 文档目的

本文用于澄清软件开发中的两条基本路径，并给出工程上的适用边界：

1. 需求导向（Response-driven）
2. 问题导向（Model-driven）

目标是避免两种常见误区：

- 只追功能交付，导致结构失控
- 一上来 full DDD，导致成本过高

## 2. 两种路径的本质区分

### 2.1 需求导向（Response-driven）

输入：需求/功能列表。  
思考方式：做什么、如何最快实现。  
输出：可交付功能。

本质：

`对外部要求的响应。`

### 2.2 问题导向（Model-driven）

输入：现实问题及其约束。  
思考方式：系统本质是什么、核心对象与规则是什么。  
输出：模型 + 从模型推导出的能力。

本质：

`对问题结构的刻画。`

一句话：

`需求导向在解题，问题导向在建题。`

## 3. 与数据库设计的统一解释

数据库设计并不是类比，而是问题导向思维的典型工程形态：

1. 概念模型（Conceptual Model）：实体、关系、业务语义
2. 逻辑模型（Logical Model）：表结构、键关系、约束
3. 物理模型（Physical Model）：索引、存储、性能优化

对应到软件架构可映射为：

- 概念模型 -> Domain（领域模型）
- 逻辑模型 -> Application + Repository 抽象
- 物理模型 -> Infrastructure（数据库实现）

关键洞察：

`数据库设计从来不是“根据功能建表”，而是“根据业务结构建模”。`

## 4. 为什么数据库更容易“问题导向”

数据库属于强约束领域：

- 不建模会立刻出现冗余、冲突、不可维护
- 结构问题会在数据层被放大

而应用开发很多时候是弱约束：

- 先拼功能系统也能跑
- 结构问题不会立刻暴露

所以大量项目会滑向：

`功能堆积 > 模型构建`

## 5. 统一原则

可将软件系统统一定义为：

`对问题世界的结构化映射`

通用路径：

`现实问题 -> 抽象模型 -> 结构映射 -> 技术实现`

反模式路径：

`需求 -> 功能 -> 代码`

其风险在于缺少模型层，导致：

- 规则分散
- 语义漂移
- 系统难演进

## 6. 适用边界与建模深度

两种路径不是对错关系，而是适用范围不同。

### 6.1 需求导向更适合

- 简单 CRUD 系统
- 工具型系统
- 一次性需求
- 业务规则弱、生命周期短

### 6.2 问题导向更适合

- 业务复杂度高
- 规则与状态多
- 生命周期长
- 多团队协作与长期演进

成熟实践应采用：

`先判断问题复杂度，再决定建模深度。`

## 7. 工程判断原则（可落地）

在方案设计前先回答四个问题：

1. 规则复杂度是否持续上升？
2. 状态与生命周期是否明显增加？
3. 语义一致性是否成为团队协作瓶颈？
4. 系统是否需要长期演进与可替换？

若多数为“是”，应从需求导向升级到问题导向。

## 8. 对 DDD/Clean 的关系定位

- DDD：定义问题核心
- Clean：保护核心不被外层反向塑形

因此，问题导向不是“重方法”，而是“先稳定结构语义，再稳定功能实现”。

## 9. 结论

能力来自实现，结构来自建模。  
功能可以快速产生，但结构不稳定会吞噬后续效率。  
只有结构稳定，能力才能持续演进。

一句话原则：

`先判断复杂度，再选择建模深度；复杂系统优先问题导向。`
