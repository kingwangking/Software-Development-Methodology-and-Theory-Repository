# 最少条件方法（Minimal Conditions Method）

## 1. 文档目的

本文定义一种可复用的问题解决方法：在尽量少的外部条件下，从问题本身出发，推导可靠且可迁移的解决方案。

目标不是拒绝工具，而是避免“工具先行、方法缺位”，确保能力持续增强。

## 2. 核心定义

最少条件方法是指：

- 先移除非必要假设与依赖
- 在最基础约束下验证问题是否可解
- 由简到繁逐步引入工具与条件

一句话：

`先用最少条件想清楚，再用合适工具做得更快。`

## 3. 方法原则

### 3.1 回归问题本身

先明确问题是什么、边界是什么、成功标准是什么，而不是先选工具。

### 3.2 工具从属于方法

工具用于放大方法效果，不替代问题建模与推理能力。

### 3.3 由简到繁

从最粗粒度、最可验证方案起步，逐步引入复杂条件。

### 3.4 强调可迁移性

优先沉淀跨场景可复用的思路、约束和判定标准。

## 4. 标准执行流程

```text
Problem
-> Define Minimal Assumptions
-> Build Simplest Viable Path
-> Verify
-> Add Conditions Incrementally
-> Abstract Reusable Rule
```

步骤说明：

1. 问题澄清：写清现象、目标、约束。
2. 最少假设：保留必须条件，移除可选条件。
3. 最简解法：先做可运行、可验证的最小方案。
4. 逐步增强：每次只增加一类条件并验证影响。
5. 规则沉淀：提炼为可复用原则/模板。

## 5. 在软件开发中的应用

### 5.1 工程实现

- 无框架先验证核心逻辑
- 无自动化先手动跑通关键路径
- 无复杂架构先实现最小可用模块

### 5.2 环境问题

- 先确认最小运行依赖（runtime + 必要配置）
- 再加数据库、缓存、消息队列等扩展依赖
- 每加一层做一次可运行性验证

### 5.3 AI 协同场景

- 先由人定义问题与边界
- 再让 AI 做信息转换（澄清、对比、枚举）
- 最终由人判定并沉淀可复用结论

## 6. 可靠性与通用性来源

最少条件方法之所以可靠，是因为它降低了“环境偶然性”的干扰；
之所以通用，是因为它依赖的是问题建模与推理，而非特定工具链。

## 7. 沉淀机制（从方法到资产）

每次完成问题后，至少记录三项：

1. 最少条件集合（解题前提）
2. 最小可行路径（执行过程）
3. 可复用规则（抽象结论）

当一个结论可回答“以后这类问题该怎么处理”时，即可沉淀为工程资产。

## 8. 常见误区

1. 把“最少条件”误解为“拒绝工具”。
2. 只做极简尝试，不做后续增强。
3. 只记录结果，不记录推理过程。
4. 在问题未收敛时过早抽象。

## 9. 结论

最少条件方法不是降低标准，而是先建立稳定认知地基，再叠加效率工具。

它帮助我们在复杂环境中保持独立思考、提升解题可靠性，并形成可持续复用的能力资产。
